Проект состоит из небольшого
ядра на чистом Python, заранее построенных индексов в формате Protocol Buffers и
тонкого веб-слоя на Flask, обслуживающего одностраничный интерфейс пользователя

## Цели и область применения

В качестве коллекции используется срез
русскоязычной Википедии (`wikimedia/wikipedia`, снапшот `20231101.ru`),
упакованный в protobuf-файл; типичный размер рабочей коллекции — порядка
10<sup>5</sup> документов.

Функциональные акценты:

- булева алгебра запросов (`AND`, `OR`, `NOT`) с приоритетами и скобками;
- поиск по логическим полям документа (`title`, `text`);
- операторы соседства (`NEAR`) поверх позиционного индекса;
- ранжирование по сглаженной схеме TF–IDF с возможностью переключения на
  BM25 с весами полей и бонусом за близость термов (см. раздел
  «Ранжирование результатов»);
- веб-интерфейс с подсветкой найденных термов и отдельная страница документа.

## Архитектура

Логически система делится на три слоя.

1. **Слой хранения** (`data/`). Содержит схему Protocol Buffers
   `wikipedia_dataset.proto`, сгенерированный модуль `wikipedia_dataset_pb2.py`,
   JSON-демо-набор `documents.json`, а также — на этапе исполнения — заранее
   построенные бинарные файлы коллекции и индексов (`documents_100k`,
   `wikipedia_100k_delta_index_{text,title}.pb`,
   `wikipedia_100k_pos_index_{text,title}.pb`). Эти бинарные артефакты в
   репозиторий не коммитятся (см. `.gitignore`) и должны быть сгенерированы
   локально; загрузка в память выполняется в `data/data_loader.py`.
2. **Поисковое ядро** (`dev/`). Описывает три индексные структуры, AST
   запросов, парсер, посписки (`posting_list`), конфигурацию ранжирования и
   собственно движок (`search_engine.SearchEngine`).
3. **Веб-слой** (`app.py`, `templates/`). Flask-приложение, которое при старте
   инстанцирует `SearchEngine`, загружает коллекцию документов и обслуживает
   HTML-страницы и JSON-эндпоинты.

Ключевые классы и их назначение:

| Модуль                  | Класс / функция                            | Назначение                                                                 |
|-------------------------|--------------------------------------------|----------------------------------------------------------------------------|

## Формат данных и индексов

Схема Protocol Buffers описана в `data/wikipedia_dataset.proto` и содержит
пять сообщений:

- `Document` — атомарная единица коллекции (`id`, `title`, `text`, `url`);
- `DocumentCollection` — отображение `id2document: map<int32, Document>`;
- `PostingList` — дельта-кодированный постинг-список (`first_id`,
  `repeated deltas`);
- `InvertedIndex` — `term2postingList: map<string, PostingList>`;
- `PositionalPostingList` / `PositionalPosting` / `PositionalIndex` —
  аналогичные структуры для хранения позиций термов внутри документов.

Дельта-кодирование применяется к отсортированным спискам: хранится первый
элемент и последовательность попарных разностей, что сокращает объём и
позволяет потоковое восстановление. Восстановление выполняет функция
`delta_decode` в `dev/index.py`.

Индексы раздельны по полям: отдельные файлы содержат постинги для `text` и
`title`. При запросе без указания поля `ReverseIndex.get` объединяет постинги
обоих полей (операция объединения с последующей сортировкой), что эквивалентно
поиску «где угодно в документе».

## Конвейер индексирования

Построение артефактов выполняется скриптом `dev/make_dataset.py`. Основная
точка входа — `make_dataset(size='100k')`. Она последовательно:

1. Загружает предварительно сохранённый на диск срез русскоязычной Википедии
   (`datasets.load_from_disk("data/wikipedia_ru_100k")`). Функция
   `save_dataset` умеет скачивать этот срез из `wikimedia/wikipedia` в потоковом
   режиме.
2. Сериализует коллекцию документов в `data/documents_100k`
   (`save_documents_to_protobuf`).
3. Для каждого поля `field ∈ {text, title}`:
   - строит инвертированный индекс (`create_inverted_index_with_delta`) и
     сохраняет его как `wikipedia_100k_delta_index_{field}.pb`;
   - строит позиционный индекс (`create_positional_index`) и сохраняет его
     как `wikipedia_100k_pos_index_{field}.pb`.

Токенизация реализована через `nltk.tokenize.WordPunctTokenizer`. Текст поля
приводится к нижнему регистру; пунктуационные токены (`string.punctuation`)
отбрасываются. Инвертированный индекс хранит для каждого терма отсортированный
список уникальных `doc_id`, позиционный — упорядоченный список позиций в
пределах одного документа.

Сгенерированные файлы расположены в `data/`, их пути жёстко прошиты в
конфигурационные константы `REVERSE_INDEX_CONFIG_PATH`,
`POS_INDEX_CONFIG_PATH`, `DATA_INDEX_CONFIG_PATH` модуля
`dev/search_engine.py`. При необходимости эти параметры можно переопределить
в конструкторе `SearchEngine`.

## Язык запросов

Грамматика реализована в `QueryParser` (файл `dev/query_parser.py`) и
описывает рекурсивный спуск со следующими правилами приоритета (по убыванию):
`NOT` > `AND` > `OR`. Скобки `(` `)` позволяют переопределять приоритет.

Поддерживаются:

- **одиночные термы**: `литва`;
- **булевы операторы**: `AND`, `OR`, `NOT` (в верхнем регистре);
- **поля документа**: `field::term`, где `field ∈ {title, text}`. Если поле
  не указано, используется значение по умолчанию `text`;
- **оператор соседства NEAR**: `term1 term2 ... @k`, который требует, чтобы
  все перечисленные термы встречались в поле в пределах окна шириной `k`
  позиций. Если `@k` опущен, то `k` равно количеству термов во фразе;
- **отрицание с AND**: форма `A AND NOT B` порождает специализированный узел
  `AndWithPositivesAndNegativesNode`, позволяющий выполнить пересечение
  только по позитивным операндам, а затем вычесть постинг отрицаемого (см.
  «Алгоритмы исполнения»).

Токенизация запросов выполняется регулярным выражением
`(\(|\)|AND|OR|NOT|\w+\:\:\w+|\w+|@\d+)`; все термы приводятся к нижнему
регистру, ключевые слова (`AND`, `OR`, `NOT`) остаются в исходном виде.

Примеры валидных запросов:

```text
литва
литва AND россия
литва OR варшава
литва AND NOT россия
title::литва
title::партия труда @2
title::литва AND частично признанное государство
(литва OR эстония) AND NOT россия
```

Пустой запрос и синтаксически некорректные конструкции приводят к
`ValueError` на этапе парсинга.

## Алгоритмы исполнения запросов

Метод `SearchEngine.execute` рекурсивно интерпретирует AST, возвращая
`PostingList` — упорядоченный по возрастанию список `doc_id`-кандидатов.

- **`TermNode`** разрешается чтением постинга из `ReverseIndex.get(term, field)`.
- **`OrNode`** (`execute_or`) выполняет слияние отсортированных постингов с
  использованием min-кучи: из каждого постинга извлекается текущий минимум,
  дубликаты подавляются, что даёт линейную по суммарной длине постингов
  сложность.
- **`AndNode`** (`execute_and`) реализует многоходовое пересечение методом
  «наименьшего максимума»: на каждом шаге все курсоры продвигаются до
  максимального из текущих значений с помощью `advance(target)`; равные
  значения попадают в результирующий список.
- **`AndWithPositivesAndNegativesNode`** вычисляет пересечение всех
  позитивных операндов, затем — пересечение подвыражений под `NOT`, после
  чего применяется `execute_and_not`: двухуказательное вычитание с
  пропусканием совпадающих идентификаторов.
- **`NotNode`** в отрыве от конъюнкции не поддерживается: попытка исполнить
  «голый» `NOT` приводит к `ValueError("Not is not implemented.")`.
  Практически это означает, что в каждом `AND`-подзапросе обязан присутствовать
  хотя бы один позитивный операнд. Двойное отрицание `NOT NOT X` сворачивается
  в `X`.
- **`NearNode`** (`execute_near`) сначала сужает кандидатов через `AND`, а
  затем для каждого оставшегося документа проверяет условие соседства
  методом `near_in`: одновременно продвигается «окно» по спискам позиций
  всех термов, и документ принимается, если существует такая комбинация
  позиций `p_1, ..., p_n`, что `max(p_i) - min(p_i) + 1 <= k`.

Все `PostingList` используют явные курсоры и операции `next`, `peak`,
`advance`, `reset`, что позволяет переиспользовать постинги в разных
поддеревьях без повторного чтения с диска.

## Ранжирование результатов

`SearchEngine.search(query)` после исполнения AST собирает список позитивных
термов запроса (`collect_positive_terms`) и для каждого найденного документа
вызывает `_score_document`. Результаты сортируются по убыванию скоринга, при
равенстве — по возрастанию `doc_id` (в строковом виде), после чего из
`app.py` отдаются первые 10.

В репозитории реализованы две формулы ранжирования.

### TF–IDF (текущая активная формула)

В текущей версии `_score_document` возвращает значение `_tf_idf_score`. Для
каждого позитивного терма `t` и соответствующего поля `field` (поле терма,
либо `text` по умолчанию) вклад в итоговый скоринг равен:

$$
\mathrm{idf}(t) = \log \frac{N + 1}{df(t) + 1} + 1,
$$

$$
\mathrm{tfw}(t, d) = 1 + \log\bigl(tf(t, d)\bigr), \qquad tf(t, d) > 0,
$$

$$
\mathrm{score}(d, q) = \sum_{t \in q^{+}} \mathrm{tfw}(t, d) \cdot \mathrm{idf}(t),
$$

где `N` — число документов в коллекции (`DataIndex.total_documents`), `df(t)`
— число документов, содержащих терм `t` в указанном поле, `tf(t, d)` —
количество вхождений терма в документ, равное длине позиционного постинга.
Множество `q^{+}` — термы запроса, не находящиеся под действием `NOT`
(собираются `collect_positive_terms`). Термы с `tf(t, d) = 0` исключаются из
суммы.

Формула устойчива к редким термам (аддитивное сглаживание `+1` в IDF и `+1`
в TF-весе), не даёт деления на ноль и легко вычисляется поверх уже
загруженных индексов.

### BM25 с весами полей и бонусом за соседство (опциональная формула)

Модуль `SearchEngine` содержит альтернативную реализацию с классической
формулой Okapi BM25 и добавкой за близость термов. Она отключена «в один
комментарий» в `_score_document`; чтобы её активировать, достаточно
вернуть `self._bm25_score(doc_id, ...) + self._proximity_bonus(doc_id, ...)`.

BM25 для одного терма в одном поле вычисляется в `_bm25_term_field_score`:

$$
\mathrm{score}_{\mathrm{BM25}}(t, d, f) = \mathrm{idf}(t, f) \cdot
\frac{tf(t, d, f) \cdot (k_1 + 1)}{tf(t, d, f) + k_1 \bigl(1 - b + b \cdot dl(d, f) / \mathrm{avgdl}(f)\bigr)},
$$

где `dl(d, f)` — длина поля `f` документа `d` в токенах (рассчитывается по
позиционному индексу), `avgdl(f)` — средняя длина поля по коллекции (с
кешированием), `idf(t, f)` — сглаженная RSJ-форма с клиппингом в 0:

$$
\mathrm{idf}(t, f) = \max\!\left(\log\!\left(\frac{N - df + 0.5}{df + 0.5} + 1\right),\ 0\right).
$$

`_bm25_score` агрегирует `BM25` по полям с весами из `ScoreConfig`
(по умолчанию `title: 2.0`, `text: 1.0`): если терм запроса связан с
конкретным полем (`title::foo`), используется только это поле; иначе
суммируются взвешенные вклады всех полей.

## Подсветка совпадений и сниппеты

В JSON-ответе `/search` для каждого документа формируется короткий сниппет —
первые 200 символов поля `text` с многоточием. Это срез без учёта позиций
совпадений; более интеллектуальное извлечение фрагментов в текущей версии
не реализовано.

Подсветка совпадений выполняется на стороне клиента. На странице выдачи
(`templates/index.html`) JavaScript парсит исходный запрос приближённым
регулярным выражением, отбрасывает операторы, отрицаемые термы и поля, и
оборачивает найденные вхождения в `<mark class="hl">`. На странице документа
(`templates/document.html`) набор подсвечиваемых термов формируется уже
сервером: маршрут `/doc/<id>` пропускает параметр `q` через `QueryParser`,
вызывает `help.helper._positive_terms`, исключая термы под `NOT`, и передаёт
результат в шаблон в виде JSON.

## Веб-интерфейс и HTTP API

Приложение `app.py` создаёт единственный экземпляр `SearchEngine`, загружает
документы через `load_documents` и регистрирует следующие маршруты Flask.

### HTTP-маршруты

| Метод  | Путь                 | Назначение                                                                 |
|--------|----------------------|----------------------------------------------------------------------------|
| GET    | `/`                  | Главная страница с формой поиска (`templates/index.html`)                  |
| POST   | `/search`            | Поиск по JSON-телу `{"query": "..."}`; возвращает топ-10 результатов       |
| GET    | `/search`            | Возврат формы (передаваемой по `?q=`); шаблон `search.html` в репозитории отсутствует — этот путь рассчитан на расширение |
| POST   | `/api/documents`     | Заглушка для добавления документа; см. «Ограничения»                        |
| GET    | `/api/stats`         | Агрегированная статистика индекса                                          |
| GET    | `/doc/<doc_id>`      | Страница документа с подсветкой позитивных термов из параметра `?q=`       |

Формат ответа `POST /search`:

```json
{
  "query": "title::литва AND москва",
  "total_results": 17,
  "results": [
    {
      "id": "123",
      "score": 4.821,
      "title": "Литва",
      "url": "https://ru.wikipedia.org/...",
      "content": "Полный текст документа...",
      "snippet": "Первые 200 символов..."
    }
  ]
}
```

Поле `total_results` содержит число документов, удовлетворяющих булевой
формуле (до ограничения сверху), массив `results` — первые десять после
ранжирования.

Формат ответа `GET /api/stats`:

```json
{
  "total_documents": 100000,
  "total_terms": 1234567,
  "index_size": 9876543
}
```

Здесь `total_terms` — число уникальных термов в объединении словарей полей
`text` и `title`, а `index_size` — суммарная длина всех постинг-списков.

### Пользовательский интерфейс

`templates/index.html` предоставляет одностраничный поиск с тёмной и светлой
темой (переключатель с сохранением в `localStorage`), сворачиваемой справкой
по синтаксису, набором быстрых «чипов» с примерами и асинхронным вызовом
`/search`. Карточка результата содержит заголовок (ссылку на
`/doc/<id>?q=<исходный запрос>`), `id`, рассчитанный `score`, внешний URL
оригинала и сниппет. После отрисовки клиент подсвечивает позитивные термы.

`templates/document.html` отображает полный текст документа, метаданные
(`id`, `url`) и на клиенте подсвечивает термы, переданные сервером в
`data-highlight`.

## Установка и запуск

### Предварительные требования

- Python 3.9+.
- Собранные бинарные артефакты данных в каталоге `data/`:
  `documents_100k`, `wikipedia_100k_delta_index_text.pb`,
  `wikipedia_100k_delta_index_title.pb`,
  `wikipedia_100k_pos_index_text.pb`,
  `wikipedia_100k_pos_index_title.pb`. Эти файлы не хранятся в репозитории
  и должны быть построены локально (см. ниже).

### Установка зависимостей

```bash
pip install -r requirements.txt
```

Файл `requirements.txt` перечисляет только `Flask==2.3.3` и `protobuf`, так
как этого достаточно для исполнения готовых индексов. Для загрузки YAML-
конфигурации BM25 дополнительно требуется `PyYAML`; для пересборки индексов
— `datasets`, `nltk` и, опционально, компилятор `protoc` (если нужно
регенерировать `wikipedia_dataset_pb2.py`). Эти пакеты полезно устанавливать
явно, например:

```bash
pip install PyYAML datasets nltk
```

### Построение данных и индексов

Схема protobuf уже скомпилирована в `data/wikipedia_dataset_pb2.py`. При
необходимости её можно перекомпилировать:

```bash
protoc --python_out=. data/wikipedia_dataset.proto
```

Построение коллекции и индексов на 100 тысяч документов:

```bash
python -c "import dev.make_dataset as md; md.save_dataset(100_000); md.make_dataset('100k')"
```

Команда последовательно скачает срез `wikimedia/wikipedia` (`20231101.ru`),
сохранит его на диск, сериализует документы в protobuf и построит два
инвертированных и два позиционных индекса. Подробнее см. `data/README.md`.

### Запуск веб-приложения

```bash
python app.py
```

Flask стартует в режиме отладки на `0.0.0.0:8080`. Открыть в браузере:
<http://127.0.0.1:8080/>. Главная страница сразу выдаёт в шапке статистику
индекса (вызов `/api/stats`) и позволяет отправлять запросы.

## Конфигурация

Параметры ранжирования вынесены в `configs/score.yaml`:

```yaml
bm25:
  k1: 1.2
  b: 0.75
  field_weights:
    title: 2.0
    text: 1.0

proximity:
  enabled: true
  window: 8
  weight: 0.25
  field: text
```

Файл загружается через `ScoreConfig.load(file_path='configs/score.yaml')`.
При отсутствии файла или ошибке парсинга используются значения по умолчанию
из `BM25Config` и `ProximityConfig` (`dev/score_config.py`). Параметры
доступны в `SearchEngine.config` и применяются только при активной
формуле BM25 (см. «Ранжирование результатов»).

Пути к индексам можно переопределить, передав соответствующие словари
(`REVERSE_INDEX_CONFIG_PATH`, `POS_INDEX_CONFIG_PATH`,
`DATA_INDEX_CONFIG_PATH`) в конструктор `SearchEngine`.

## Тестирование

Интерактивные проверки (на Jupyter-ноутбуках) находятся в каталоге
`dev/test/`: `test_index.ipynb`, `test_query_parser.ipynb`,
`test_boolean_op.ipynb`, `test_execute.ipynb`, `test.ipynb`. Каталог
`dev/test/` перечислен в `.gitignore` и в публичный репозиторий может не
попадать; при локальной разработке ноутбуки удобно использовать для
регрессионной проверки парсера и алгоритмов пересечения постингов.

Автоматизированного pytest-сьюта в текущей версии нет. В качестве простейшей
«дымовой» проверки после запуска сервера можно выполнить:

```bash
curl -s -X POST http://127.0.0.1:8080/search \
  -H 'Content-Type: application/json' \
  -d '{"query": "литва AND NOT россия"}' | head -c 400
curl -s http://127.0.0.1:8080/api/stats
```

Альтернативно — программный вызов движка без HTTP:

```python
from dev.search_engine import SearchEngine
engine = SearchEngine()
from data.data_loader import load_documents
load_documents(engine=engine, file='data/documents_100k')

hits = engine.search('title::литва AND частично признанное государство')
print(len(hits), hits[:3])
```

## Ограничения текущей реализации

Ниже перечислены ограничения, подтверждённые прямым чтением кода.

- **`/api/documents` нефункционален.** Обработчик `add_document` в `app.py`
  вызывает `engine.add_document(...)`, но такого метода в `SearchEngine` нет.
  Любой POST-запрос на этот эндпоинт приведёт к ошибке 500. Инкрементальное
  добавление документов в текущей архитектуре требует перестроения
  protobuf-индексов.
- **`NOT` вне конъюнкции не поддержан.** Парсер принимает запрос вида
  `NOT term`, но исполнение `NotNode` вне `AndWithPositivesAndNegativesNode`
  завершается исключением. Практически каждый `NOT` должен сопровождаться
  хотя бы одним позитивным операндом.
- **GET `/search` ссылается на несуществующий шаблон `search.html`.** Этот
  путь оставлен как заглушка для будущего расширения; интерактивный поиск
  работает через POST-вызов из `templates/index.html`.
- **Активная формула ранжирования — TF–IDF.** Несмотря на полностью
  реализованные BM25 и бонус за соседство, вызов `_score_document`
  возвращает только `_tf_idf_score`. Переключение делается однострочным
  изменением; см. «Ранжирование».
- **Сниппет не контекстен.** В выдаче используются первые 200 символов
  поля `text` без учёта позиций совпадений. Подсветка на карточке результата
  выполняется только после отрисовки, на клиенте.
- **Поддерживается лексическое совпадение, не морфология.** Токены
  нормализуются исключительно приведением к нижнему регистру. Стемминг,
  лемматизация, стоп-листы, исправление опечаток и wildcard не реализованы.
- **Индексы читаются целиком в память** при старте (`ParseFromString` поверх
  всего файла). Для существенно бо́льших коллекций потребуется разделение на
  сегменты или mmap-подход.
- **Пересборка коллекции несёт зависимости уровня ML-экосистемы**
  (`datasets`, `nltk`). В `requirements.txt` они не указаны намеренно, чтобы
  не утяжелять продуктовый запуск.
